# Zenn記事 執筆ルール

このドキュメントは、Zennブログ記事を執筆する際の標準的なルールとスタイルガイドをまとめたものです。

## フロントマター

記事の冒頭には必ずYAML形式のフロントマターを記載します。

```yaml
---
title: "記事のタイトル"
emoji: "🔥"
type: "tech" # tech: 技術記事 / idea: アイデア
topics: ["トピック1", "トピック2", "トピック3"]
published: true
---
```

### フロントマターの各項目

- **title**: 記事のタイトル。具体的で魅力的なものにする
- **emoji**: 記事を表す絵文字1つ
- **type**:
  - `tech`: 技術記事
  - `idea`: アイデア記事
- **topics**: 記事に関連するトピックを配列で指定（通常3つ程度）
- **published**:
  - `true`: 公開
  - `false`: 下書き

## 記事構成

### 基本構造

1. **はじめに** （必須）
   - 記事の背景や目的を説明
   - 読者が得られる価値を明示
   - 前提知識や参考記事へのリンク

2. **3行でまとめ** （推奨）
   - 記事の要点を3つの箇条書きで簡潔にまとめる
   - 読者が内容を素早く把握できるようにする

3. **本文**
   - 明確なセクション構造（`##` で見出し）
   - サブセクションは `###` を使用
   - 必要に応じて `####` も使用可能

4. **おわりに** （必須）
   - 記事の総括
   - 今後の展望や感想
   - 読者への感謝の言葉

### セクションの命名規則

- 明確で分かりやすい日本語の見出し
- 内容を端的に表現する
- 階層構造を適切に使用

## 画像の扱い

### 画像の配置

- 画像は `/images/<記事ID>/` ディレクトリに配置
- 記事IDはフロントマターのファイル名から取得

### 画像の参照

```markdown
![](/images/<記事ID>/filename.png)
```

- altテキストは必要に応じて記載
- 画像の説明が必要な場合は、画像の下に *説明文* の形式で追加

### 画像の命名

- 分かりやすいファイル名を使用
- 英数字とアンダースコア、ハイフンを使用
- 例: `sample1.png`, `qwen3.png`, `architecture_diagram.png`

## リンクの扱い

### 外部リンク

通常のマークダウンリンクまたは直接URL記載：

```markdown
https://example.com
```

### 参考文献・脚注形式

学術論文などの参考文献は脚注形式で記載：

```markdown
DeepSeek-R1^[[DeepSeek-R1: Incentivizing Reasoning Capability in LLMs via Reinforcement Learning](https://arxiv.org/abs/2501.12948)]
```

### 引用元の明示

画像や図を引用する場合は、画像の直後に出典を記載：

```markdown
![](/images/<記事ID>/figure.png)
*論文タイトル (URL)*
```

## コンテンツ要素

### コードブロック

言語を指定してシンタックスハイライトを適用：

````markdown
```python
def hello():
    print("Hello, World!")
```
````

### 数式

LaTeX形式で記載：

```markdown
$$
R(\tau) = \begin{cases}
-1 & \text{if format is incorrect} \\
\text{compare}(\text{patch}_{\text{pred}}, \text{patch}_{\text{gt}}) & \text{otherwise}
\end{cases}
$$
```

インライン数式は `$...$` を使用：

```markdown
状態 $s_t$ から観測 $o_t=O(s_t)$
```

### 箇条書き

- 通常の箇条書きは `-` を使用
- 順序付きリストは `1.`, `2.`, ... を使用
- ネストも適宜使用

### 強調

- **太字**: `**テキスト**` - 重要な用語やキーワード
- *イタリック*: `*テキスト*` - 補足説明や引用元

### 折りたたみセクション

長いコードやプロンプトは折りたたみ可能にする：

```markdown
:::details プロンプト
長いコンテンツ...
:::
```

### テーブル

比較や整理にはテーブルを活用：

```markdown
| 項目 | 説明 |
|------|------|
| A    | 説明A |
| B    | 説明B |
```

## 文体とスタイル

### 基本方針

- 丁寧な日本語を使用（です・ます調）
- 専門用語は適切に説明
- 読者の理解レベルを考慮した説明
- 具体例や図解を活用

### 用語の扱い

- 英語の技術用語は原語のまま使用（例: Reinforcement Learning）
- 日本語訳を併記する場合: 強化学習（Reinforcement Learning、RL）
- 初出時に定義や説明を記載
- 略語は初出時にフルスペルを記載

### 文章の構成

- 1段落1トピックを基本とする
- 長い文章は適切に分割
- 箇条書きを活用して読みやすくする
- 論理的な流れを意識

## 日本語表現と略称のルール

### 技術用語の初出表記

技術用語を初めて使用する際は、以下のパターンで表記します：

#### パターン1: 日本語訳（英語フルスペル、略称）

```markdown
エージェント型強化学習（Agentic Reinforcement Learning、Agentic RL）
モンテカルロ木探索（Monte Carlo Tree Search、MCTS）
視覚言語モデル（Vision-Language Model、VLM）
```

#### パターン2: 日本語訳(略称)

よく知られた技術用語や文脈が明確な場合：

```markdown
人間のフィードバックによる強化学習(RLHF)
教師ありファインチューニング(SFT)
Chain-of-Thought(CoT)プロンプティング
```

#### パターン3: 英語^[リンク]形式

論文名などで脚注リンクを付ける場合：

```markdown
DeepSeek-R1^[[DeepSeek-R1: Incentivizing Reasoning Capability](https://arxiv.org/abs/XXX)]
```

### 略称の2回目以降の使用

- 初出で定義した略称は、2回目以降はそのまま使用
- 例: RL、CoT、RAG、VLM、MCTS、SFT など
- 文脈が明確な場合は日本語訳のみも可

```markdown
# 初出
エージェント型強化学習（Agentic Reinforcement Learning、Agentic RL）

# 2回目以降
Agentic RLでは...
または
エージェント型強化学習では...
```

### カタカナ表記の原則

#### 使用可能なカタカナ

技術分野で定着している用語：

- プロンプト、アルゴリズム、モデル、データ、システム
- ファインチューニング、ベンチマーク、フレームワーク
- エージェント、パラメータ、アーキテクチャ

#### 避けるべきカタカナ

意味が不明確になる外来語は日本語化：

- ✗ インプット → ○ 入力
- ✗ アウトプット → ○ 出力
- ✗ インプルーブ → ○ 改善

### 動詞の使い分け

#### 研究・開発に関する動詞

```markdown
- 提案する、提示する
- 導入する、採用する
- 実施する、実行する、行う
- 試みる、試す
- 実現する、可能にする
- 達成する、向上させる
- 示す、明らかにする
- 学習する、最適化する
```

#### 例

```markdown
✓ この手法では、報酬モデルを取り除いて学習コストを下げる具体策を示しました
✓ RLにより、エージェントは試行錯誤から学び取れるようになります
✓ この研究では、複数の最適化手法を導入しています
```

### 文末表現

#### 断定を避ける表現

研究内容や一般的な事実を述べる際：

```markdown
- 〜といえます
- 〜と考えられています
- 〜が期待されます
- 〜と位置付けられます
- 〜の可能性があります
```

#### 実験結果を述べる表現

```markdown
- 〜を示しています
- 〜が確認されました
- 〜を達成しています
- 〜となっており
- 〜が見受けられます
```

#### 説明・定義の表現

```markdown
- 〜です
- 〜ます
- 〜できます
- 〜する必要があります
```

**例:**

```markdown
✓ この手法が有効であることを示しています（実験結果）
✓ RLが重要な役割を果たしているといえます（分析）
✓ エージェントは環境と対話しながら学習します（説明）
```

### 接続表現

#### 対比・逆接

```markdown
- 一方（で）、しかし、ただし
- これに対し、それに対して
```

#### 追加・並列

```markdown
- また、さらに、加えて
- 同様に、同時に
```

#### 因果・結果

```markdown
- このため、そのため、これにより
- 結果として、その結果
- したがって、よって
```

#### 例示・具体化

```markdown
- 例えば、具体的には
- 特に、とりわけ
```

**例:**

```markdown
✓ 従来手法では精度が低い。一方、提案手法では大幅に改善されています。
✓ この手法により、学習コストを削減できます。さらに、性能も向上しています。
✓ 複数のベンチマークで評価しました。具体的には、MATH500とGSM8Kを使用しています。
```

### 引用・説明での助詞の使い方

#### 場所・範囲を示す

```markdown
- 〜では（研究・論文）
- 〜において（状況・条件）
- 〜に対して（対象）
- 〜について（テーマ）
```

#### 方法・手段を示す

```markdown
- 〜を用いて、〜を使って
- 〜により、〜によって
- 〜を通じて
```

#### 定義・呼称を示す

```markdown
- 〜と呼ばれる
- 〜という
- 〜のような
- 〜といった
```

**例:**

```markdown
✓ DeepSeek-R1では、GRPOというRLアルゴリズムを用いています
✓ 数学タスクにおいて、大幅な性能向上を達成しています
✓ ツール利用と呼ばれる機能について説明します
```

### 数値・単位の表記

#### 数値の表記

```markdown
- パーセント: 36.0%、73.6%
- スコア: Score: X.XXX
- パラメータ数: 14B、671B
- 件数: 500件、460万件
```

#### 順位の表記

```markdown
- 1位、2位、3位（算用数字）
- 第1位、第2位（「第」を付ける場合も）
- Private LB Xth（英語表記の場合）
```

### 記号の使い方

#### 括弧の使い分け

```markdown
- 全角（）: 日本語の補足説明
- 半角(): 英語略称、数式、プログラムコード
```

**例:**

```markdown
✓ エージェント型強化学習（Agentic RL）
✓ 報酬関数 R(s, a)
✓ Python の dict() 関数
```

#### 中黒・読点

```markdown
- 中黒「・」: 並列する名詞を列挙
- 読点「、」: 文中の区切り
```

**例:**

```markdown
✓ 計画・推論・ツール利用・記憶保持の能力
✓ モデルを学習し、評価し、改善します
```

### よく使う表現パターン

#### 研究紹介の定型表現

```markdown
- 〜が発表されました
- 〜を提案しています
- 〜に焦点を当てています
- 〜を目的としています
- 〜を実証しています
```

#### 結果・効果の表現

```markdown
- 性能が向上する、改善する
- 〜を実現する、可能にする
- 効果的である、有効である
- 〜に成功する、達成する
```

#### 課題・問題の表現

```markdown
- 課題がある、問題がある
- 限界がある、困難である
- 〜が求められる、必要である
```

## その他のルール

### HTMLコメント

下書きや一時的に非表示にしたい部分：

```markdown
<!-- ![](/images/<記事ID>/sample4.png) -->
```

### 前提知識の記載

記事の冒頭や該当セクションで前提知識を明示：

```markdown
### 本記事の前提
- PPOやGRPOといったRLアルゴリズムの解説は他の多くの記事で既に説明されているため、本記事では割愛します。
- DeepSeek-R1の研究を前提とする箇所がいくつかあります。
```

### 図表の活用

- 複雑な概念は図解で説明
- 研究論文からの引用図には必ず出典を明記
- 自作の図は分かりやすいデザインを心がける

### 区切り線

セクション間の大きな区切りには水平線を使用：

```markdown
---
```
